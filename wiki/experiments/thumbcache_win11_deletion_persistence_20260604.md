## 목적
(1) 원본 파일 변경·삭제 후 썸네일 엔트리가 잔류하며 이미지 복구가 가능한지 확인.
(2) DB가 최대용량에 도달한 상태에서 신규 이미지 열람 시 교체 알고리즘 동작 및 교체된 엔트리의 복구 가능성 파악.

## 환경
- OS: Windows 11
- 도구: Thumbcache Viewer
- DB 파일: thumbcache_2560.db (최대용량 테스트)

## 절차

### 케이스 1 — 원본 파일 변경·삭제
1. 원본 파일이 존재하는 상태에서 파일 탐색기로 폴더 열람 → 썸네일 생성 확인
2. Thumbcache Viewer로 해당 이미지 엔트리 확인 및 스크린샷 저장
3. 원본 파일 변경 후 Thumbcache Viewer에서 동일 엔트리 잔류 여부 재확인
4. 원본 파일 삭제 후 동일 엔트리 잔류 여부 재확인
5. "Save Selected"로 썸네일 이미지 추출 시도

### 케이스 2 — 최대용량 도달 후 신규 이미지 열람
1. thumbcache_2560.db를 최대용량 상태로 유지
2. 신규 이미지 9개 열람
3. Thumbcache Viewer로 이전/이후 엔트리 목록 비교
4. 삭제된 엔트리의 Cache Entry Offset 범위와 신규 엔트리의 Offset 범위 교차 분석

## 관찰 결과

### 케이스 1
- 원본 파일 변경 후 → 엔트리 **잔류** ✅
- 원본 파일 삭제 후 → 엔트리 **잔류** ✅
- "Save Selected" 이미지 추출 → **성공** ✅

### 케이스 2
9개 신규 이미지 추가 열람 → 6개 기존 엔트리 삭제:
- 삭제 기준: 신규 9개 합산 용량(~369 KB) < 삭제 6개 합산 용량(~647 KB)
- LRU(Least Recently Used) 알고리즘으로 알려져 있으나 관찰 결과는 **FIFO(First In First Out) 방식일 가능성도 있음** (삭제된 엔트리가 가장 오래된 순번 558~563번에 집중)

## 해석
- **케이스 1**: 파일 삭제 후에도 thumbcache 엔트리가 잔류하며 이미지 복구 가능. quick_2014의 주장과 일치.
- **케이스 2**: 최대용량 도달 시 오래된 엔트리가 교체됨. LRU로 알려져 있으나 삭제 패턴이 FIFO와 유사. 교체된 엔트리는 Offset 범위가 신규 엔트리와 겹치므로 **복구 어려울 것으로 예측**.
