[질문3] Manifest.db가 손상되거나 누락된 경우, 00~FF 디렉터리에 저장된 파일들만으로 어느 정도까지 복구 또는 식별이 가능한지 궁금합니다.

[답변3] 암호화되지 않은 백업이라면 00~FF 디렉터리 내 파일들의 실제 내용을 직접 확인할 수 있습니다. 다만 Manifest.db가 없으면 원래 파일 경로 정보를 확인하기 어렵습니다.

그러나 fileId가 무작위 값이 아니라 domain + "-" + relativePath 문자열에 대해 SHA-1 해시를 적용해 생성된다는 점을 활용하면, 주요 경로들의 해시값을 미리 계산하여 fileId와 대조함으로써 일부 파일의 원래 경로를 추정할 수 있습니다.

예를 들어 3d0d7e5fb2ce288813306e4d4636395e047a3d28는 HomeDomain-Library/SMS/sms.db에 해당합니다.

또한 파일 형식이나 내부 구조가 이미 알려진 경우에는 내용 분석을 통해 파일 종류를 추정할 수도 있습니다.

암호화된 백업에서도 fileId 생성 규칙 자체는 동일하게 적용되는 것을 확인했습니다. 다만 Manifest.db 손상으로 인해 File 테이블 내 래핑된 복호화 키를 얻을 수 없는 경우에는 현실적인 비용 내에서 개별 파일 복호화가 매우 어려울 것으로 보입니다. 이 경우에도 fileId 생성 메커니즘을 활용하여 어떤 경로의 파일이 존재했는지 정도는 일부 추정 가능할 것으로 생각됩니다.

[질문11] Binary plist 파싱 시 대용량 Dictionary 데이터에 대한 별도의 검색 최적화 기법이 존재하는지 궁금합니다.

[답변11] Dictionary에 특화된 검색 최적화 기법이 존재하는지는 확실하지 않습니다. 다만 전체 데이터를 대상으로 한다면 오브젝트 테이블만 우선 파싱한 뒤 특정 값의 존재 여부를 확인하는 정도의 최적화는 가능할 것으로 생각됩니다.

하지만 Binary plist 포맷은 기본적으로 오프셋 테이블을 읽고 해당 오브젝트 위치로 이동하여 데이터를 해석하는 구조이므로, 각 오브젝트를 최소 한 번은 확인해야 실제 데이터 형식을 파악할 수 있습니다.

한편 저장 공간 최적화 측면에서는 동일한 데이터가 여러 번 등장하더라도 오브젝트 테이블에는 한 번만 저장하고, 오프셋 테이블에서 동일 위치를 참조하는 방식으로 중복 저장을 줄이는 구조를 사용합니다.

다만 SQLite 데이터베이스와 같은 애플리케이션 데이터 파일이 함께 백업되는 경우에는, iOS 백업 자체의 복구 기법보다는 SQLite 복구 기법 등 개별 파일 형식에 대한 포렌식 분석 기법을 적용하는 방식으로 접근할 수 있습니다.

[질문13] iOS 백업 데이터에서 SHA-1 해시 문자열을 사용한다고 하셨는데, 서로 다른 두 파일이 동일한 해시값을 가지게 되면 같은 파일로 인식되는지 궁금합니다.

[답변13] iOS 백업에서 사용되는 SHA-1 값은 파일 내용이 아니라 domain + "-" + relativePath 문자열을 기반으로 생성됩니다.

따라서 같은 해시값이 나온다면 이는 동일한 경로를 의미합니다. 반대로 파일 내용이 완전히 동일하더라도 도메인이나 relativePath가 다르면 서로 다른 SHA-1 값이 생성됩니다.

즉, 이 해시는 파일 내용 동일성 검증보다는 경로 식별, 데이터 분산 저장, 경로명 단순화 등의 목적으로 활용됩니다.

[질문14] 암호화된 iOS 백업에서 특정 파일을 복호화하려면 어떤 정보가 필요하며, Manifest.db가 손상된 경우 어떤 문제가 발생하는지 궁금합니다.

[답변14] 사용자가 설정한 백업 암호를 이용해 Manifest.plist의 BackupKeyBag을 복호화하고, 그 정보를 바탕으로 Manifest.db 복호화 키를 획득하여 Manifest.db를 복호화합니다. 이후 File 테이블의 file 필드에 저장된 래핑된 키를 해제하여 개별 파일을 복호화합니다.

따라서 Manifest.db가 손상되어 File 테이블 내 래핑된 키 정보를 획득할 수 없는 경우에는 개별 파일 복호화가 현실적으로 매우 어려워집니다. 이 경우 fileId 생성 규칙을 활용해 일부 파일의 존재 여부나 예상 경로 정도만 추정할 수 있습니다.
