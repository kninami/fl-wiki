iOS 백업 데이터 Manifest.db 파일 구조 분석과 plist 텍스트 및 바이너리 형식에 대한 분석 내용을 설명드리겠습니다. 먼저 각 백업 데이터가 어떻게 구성되는지 간단히 설명드리고 넘어가겠습니다. 실험 환경에 대해 설명드리면, 실험 디바이스로는 iPhone 6S를 사용했고 iOS 버전은 15.8.7입니다. 백업 시 암호화 여부를 선택할 수 있는데, 제 경우에는 암호화하지 않은 iOS 백업을 대상으로 테스트를 수행했습니다. 분석 시에는 크게 세 가지 도구를 사용할 수 있습니다. 하나는 일반적인 SQLite 뷰어 프로그램, 하나는 16진수 에디터, 하나는 일반적인 텍스트 에디터입니다. 추가로 macOS에 내장되어 있는 plist 명령형 도구인 plutil을 사용해서 분석을 수행할 수 있습니다.

00:59
백업 절차를 간단히 설명드리겠습니다. 케이블을 이용해 iPhone을 연결하면 Finder를 통해 백업을 수행할 수 있습니다. Finder의 위치 탭을 보면 본인의 iPhone이 표시되고, 이 위치를 클릭하면 백업 관련 옵션 지정, 암호 설정, 백업 수행을 모두 할 수 있습니다. 백업이 완료되면 사용자 홈 폴더 아래 ~/Library/Application Support/MobileSync/Backup/ 하위 디렉터리에 각 백업마다 식별자로 해시 형태의 문자열을 이름으로 가진 디렉터리가 생성됩니다. 이 디렉터리 안에 백업 관련 데이터가 저장됩니다. 백업 데이터는 크게 파일 4개와 디렉터리 256개로 구성되어 있습니다. 첫 번째는 Manifest.db 파일입니다.

01:57
이 파일은 SQLite3 기반의 데이터베이스 파일로 iOS 10 버전부터 도입되었습니다. 이 파일에서 관리하는 데이터는 백업된 파일들의 해시, 경로 정보, 속성 정보를 담고 있는 Files 테이블과 Key-Value 형태의 데이터를 담고 있는 것으로 추정되는 Properties 테이블, 이 2개 테이블로 구성되어 있습니다. 다음은 Info.plist 파일입니다. 이 파일은 XML 기반의 plist 파일이기 때문에 일반 텍스트 에디터나 XML 파서로 열어볼 수 있습니다. 이 파일에는 백업 대상 기기에 관한 정보가 들어갑니다.

02:51
예를 들면 기기명, 일련번호, 전화번호와 같은 정보가 들어 있어 직접 확인할 수 있습니다. 추가로 백업을 수행한 호스트의 macOS 버전 정보도 같이 저장됩니다. 다음으로 Manifest.plist 파일입니다. 이 파일은 바이너리 형태의 plist 파일로, 백업과 관련된 요약 정보가 들어갑니다. 암호화 수행 여부, Lockdown 인증서 정보, 백업 대상 앱 목록 등이 저장되는데, 이러한 정보는 바이너리 plist 파서나 별도의 뷰어 프로그램, 또는 plutil 같은 명령형 도구를 통해 변환한 후 열어보셔야 합니다. 다음으로 Status.plist 파일입니다. 이 파일 역시 바이너리 형태의 plist 파일로, 백업 진행 상태와 관련된 정보를 저장합니다.

03:53
다음으로 디렉터리 256종입니다. 이 디렉터리들은 백업된 각 파일을 저장하는 역할을 하며, 백업 파일들은 00부터 FF까지 총 256개의 디렉터리에 분산 저장됩니다. 각 백업 대상 파일의 domain + '-' + relativePath 문자열을 결합하여 SHA-1 해시를 구한 뒤, 이 해시값의 앞 두 글자를 따서 해당 이름의 폴더에 파일을 저장하는 구조입니다. 이렇게 설계된 이유는 한 디렉터리 내에 수많은 파일이 들어갈 때 발생할 수 있는 파일 시스템 성능 저하를 회피하기 위함입니다.

04:42
암호화를 설정하지 않았다면 저장된 파일을 그대로 읽을 수 있습니다. 반면 암호화를 사용해서 백업했다면 해당 파일은 AES-256 CBC 모드로 암호화됩니다. 암호화에 사용된 키는 별도의 키로 래핑되어 Manifest.db의 Files 테이블 각 레코드의 file 필드에 추가됩니다. 따라서 암호화된 백업은 Manifest.db 파일이 있어야 복호화가 가능합니다. 다음은 Manifest.db 파일에 대한 분석입니다. 이 파일은 백업된 파일들에 대한 메타 정보 저장소라고 보시면 됩니다.

05:30
크게 두 가지 테이블이 있습니다. 백업된 파일들에 대한 정보를 저장하는 Files 테이블과, Key-Value 형태의 정보를 저장하기 위한 Properties 테이블입니다. 다만 제가 분석한 시점에는 Properties 테이블에 아무런 데이터가 없었고, 다른 사례를 찾아봐도 해당 테이블에 데이터가 들어 있는 경우를 확인하지 못했습니다. 따라서 주된 분석 내용은 Files 테이블을 중심으로 설명드리겠습니다. Files 테이블에는 백업된 모든 파일에 대한 정보가 저장됩니다. fileID는 domain + '-' + relativePath를 SHA-1 해시한 값으로, 00~FF 디렉터리 하위의 파일명 역할을 합니다.

06:24
그 외 주요 컬럼으로는: domain(해당 파일이 속한 앱/서비스 식별자), relativePath(기기 내 원본 경로), flags(해당 경로가 파일인지, 디렉터리인지, 심볼릭 링크인지 구분하는 플래그)가 있습니다. 마지막으로 file 필드가 있는데, 이 필드에는 마지막 수정 시각, 파일 크기, 생성 시각 등의 메타데이터가 바이너리 plist 형태로 저장됩니다. 이 메타데이터가 왜 필요한지 생각해 보면, iOS 백업에는 증분 백업 개념이 있기 때문입니다.

07:04
증분 백업은 마지막 백업 시점 이후 변경된 파일만 새로 저장하는 방식입니다. 어떤 파일이 변경되었는지 판단하려면 마지막 수정 시각과 파일 크기 같은 정보가 필요하므로, 의도적으로 file 필드에 저장하는 것이라 볼 수 있습니다. 암호화된 백업의 경우에는 각 파일을 암호화하는 데 사용된 암호키를 래핑하여 file 필드에 같이 저장합니다.

07:47
Properties 테이블은 Key-Value 형태의 데이터를 저장하기 위해 만들어 놓은 테이블로 추정되는데, 분석 시점에는 비어 있었고 다른 사례를 찾아봐도 이 테이블을 사용한 경우가 보이지 않아 이 부분은 넘어가도록 하겠습니다. 이어서 plist 텍스트 및 바이너리 형식에 대해 설명드리겠습니다. plist는 macOS나 iOS에서 구조화된 데이터를 직렬화하는 데 사용되는 포맷으로, 텍스트(XML) 형식과 바이너리(bplist) 형식 두 가지가 존재합니다. 메모리 데이터를 직렬화하는 포맷인 만큼, 한쪽 포맷에서 로딩하여 다른 포맷으로 상호 변경하는 것도 가능합니다.

08:41
XML 형식은 사람이 읽을 수 있고 수동 편집도 가능한 장점이 있고, 바이너리 형식은 파싱이 비교적 빠르고 파일 크기가 작아진다는 장점이 있습니다. iOS 백업에서 저장되는 메타데이터 파일들, 즉 앞서 언급한 3개 파일은 모두 이 형식으로 저장되어 분석할 수 있습니다. 첫 번째로 설명드릴 내용은 plist 텍스트 포맷입니다. 이 포맷은 XML 기반의 텍스트 파일이라고 보시면 됩니다. 일반적인 XML 파서로 파싱할 수 있고, 일반 텍스트 에디터로도 열어볼 수 있습니다.

09:35
plist XML은 최상위에  태그로 시작하며, 그 내부에 각 자료형에 맞는 데이터가 들어갑니다. plist 파일들을 확인해 보면 대체로 Dictionary 형태로 key-value 쌍의 구성을 가지는 것을 확인할 수 있습니다. 사용 가능한 자료형은 크게 8가지이며, 세부 내용은 DTD 문서를 참고하시면 됩니다. 다음은 plist 바이너리 포맷입니다. 크게 네 가지 요소로 구성됩니다. Header(8바이트)는 bplist 포맷과 버전을 명시합니다. Object Table은 직렬화 대상 데이터가 실제로 저장되는 가변 길이 영역입니다.

10:36
Object Table에 데이터를 기록할 때는 먼저 1바이트(또는 그 이상)로 자료형과 데이터 길이를 표기한 후, 이어서 실제 데이터를 기록하는 방식으로 모든 데이터를 직렬화합니다. 다음은 Offset Table입니다. Offset Table도 가변 길이이며, Object Table의 각 항목에 접근하기 위한 레퍼런스 역할을 합니다. Offset Table의 각 항목은 Object Table 내에서 해당 객체가 위치한 오프셋을 가리킵니다. 마지막으로 Trailer입니다.

11:26
Trailer는 바이너리 plist 파일을 열 때 가장 먼저 읽는 32바이트 영역으로, 이 정보를 통해 파일 전체 구조(Offset Table의 위치, 각 참조의 바이트 크기, 총 객체 수, 루트 객체 인덱스 등)를 파악할 수 있습니다. 다음은 바이너리 plist를 실제로 로딩하는 절차입니다. 첫 번째 단계는 Header 확인입니다. 맨 앞 8바이트를 통해 이 파일이 bplist인지, 그리고 bplist라면 몇 버전인지를 확인합니다. 두 번째는 Trailer 로딩입니다. Trailer는 파일 맨 마지막 32바이트에 위치하며, 이 영역을 통해 Offset Table의 시작 위치(offsetTableOffset), 각 항목의 바이트 단위(offsetIntSize), 총 객체 수(numObjects) 등의 전반적인 구조 정보를 먼저 확인합니다.

12:30
세 번째는 Offset Table 로딩입니다. Trailer에서 확인한 offsetTableOffset 위치로 이동한 뒤, offsetIntSize 단위로 numObjects회 끊어 읽습니다. 각 값은 Object Table 상에서 해당 객체가 위치한 오프셋이 됩니다. 네 번째는 Object Table 이동 및 값 해석입니다. 예를 들어 Offset Table의 첫 번째 항목이 가리키는 0x08 위치가 Object Table의 시작임을 확인할 수 있습니다. 꼭 그렇지 않더라도, 헤더가 8바이트이므로 그다음부터 Object Table이 시작된다고 예상할 수 있습니다.

13:28
Object Table 시작점을 읽어 보면 이 예시에서는 0xD4입니다. 0xD는 Dictionary 타입을 의미하고, 하위 니블 4는 key-value 쌍이 총 4개 존재한다는 뜻입니다. 따라서 Dictionary 마커 이후로 objectRefSize × (4 × 2) = 8바이트만큼 참조 인덱스가 나열됩니다. 이 여덟 개 값은 key, value, key, value, key, value, key, value 순서로, 각각 Offset Table에서의 인덱스입니다.

14:28
예를 들어 첫 번째 인덱스 0x01을 Offset Table에서 찾아가면 0x11이라는 오프셋 값을 얻고, Object Table의 0x11 위치로 가면 0x54를 확인할 수 있습니다. 0x5는 ASCII String 타입, 하위 니블 4는 4바이트 길이를 의미하므로, 이어지는 4바이트를 읽으면 "STRA"라는 key 값을 얻게 됩니다. 이런 식으로 인덱스 참조 → 오프셋 이동 → 타입·길이 해석 → 값 읽기를 반복하여 plist 전체를 파싱합니다.

15:31
지원하는 데이터 타입으로는 Array, Dictionary, Boolean, 정수, 실수, 문자열 등 일반적인 프로그래밍 언어 수준의 자료형을 갖추고 있으며, 세부 구현은 Apple이 공개한 소스 코드(CFBinaryPList.c 등)를 참고하시면 됩니다. 여기까지 포맷을 이해했다면, 이제 각 iOS 백업 파일을 포맷에 맞게 열어 정보를 확인하는 단계만 남습니다. Info.plist는 XML 기반의 plist 파일로, 일반 텍스트 뷰어나 XML 파서로 로딩하면 됩니다. 이 파일에는 백업 대상 단말의 전화번호, 일련번호, iOS 버전, 디바이스명 등 단말 기초 정보와 함께, 백업을 수행한 호스트 macOS의 버전 정보도 기록됩니다.

16:35
다음은 Manifest.plist입니다. 이 파일은 바이너리 기반의 plist 파일로, 백업 파일 자체에 대한 기초 정보를 주로 저장합니다. 예를 들어 암호화에 사용된 Keybag 데이터, 암호화된 백업인지 여부, 백업 수행 날짜 등 백업 전반에 관한 정보가 담겨 있습니다. 마지막으로 Status.plist 파일입니다. 이 파일 역시 바이너리 plist이며, 백업 상태 정보가 주로 저장됩니다. 백업 완료 여부, 풀백업인지 증분 백업인지 여부, 스냅샷 생성 시각 등 백업 진행 상태를 확인할 수 있습니다.
